4. 该异常被 `runAI` 的顶层 `catch` 捕获，统一按"网络/API错误"处理
5. 用户看到的是虚假的红色错误框（"网络连接问题 / Key无效 / CORS拦截"），而非真正原因

**修复内容：**

**文件：** `app.js`，`renderCard` 函数（第133行起）

| 字段 | 原来 | 现在 |
|------|------|------|
| `homework` | 直接 `.map()`，假设一定是数组 | `Array.isArray` 判断 → 数组正常遍历；字符串直接展示；对象 JSON.stringify 展示；`undefined` 显示"（无）" |
| `exam_content` | 假设一定是对象，直接 `Object.entries` | 增加类型判断，对象正常处理，字符串直接展示 |
| `course_name` | 直接使用，假设一定有值 | 兜底 `"未识别课程名"` |
| `importance` | 直接使用，假设一定有值 | 兜底 `"未知"` |

**文件：** `app.js`，`runAI` 函数（第72行起）

- 新增**第二层 try-catch**，专门捕获 `renderBoard` / `renderCard` 的渲染异常
- 渲染失败时显示**橙色警告框**（区别于红色的网络错误），包含：
  - 具体错误信息
  - 可能原因提示
  - `<details>` 折叠面板展示 AI 返回的原始 JSON（方便调试 prompt）

**现在的三层错误处理：**

### 问题 2：缺少错误处理

**原因：**
`runAI()` 函数体内没有任何 `try-catch`：

1. 如果 `fetch` 失败（网络断开、CORS 拦截、API Key 无效返回 401 等），Promise 被拒绝但无人捕获 → 浏览器控制台报 `Unhandled Promise Rejection`，界面永远停留在 `"AI解析中..."`
2. 即使 `fetch` 成功，API 可能返回非 200 状态码或异常 JSON 结构，访问 `data.choices[0].message.content` 会直接抛异常，同样无人捕获
3. 用户输入为空时也没有前置校验，会发送无意义的空请求

**修复内容：**
**文件：** `app.js`

| 改动 | 说明 |
|------|------|
| 新增空输入校验 | 在发送请求前检查 `input` 是否为空，为空时提示用户先输入内容 |
| 整个函数体包裹 `try-catch` | 捕获所有异步和同步异常 |
| 新增 HTTP 状态码检查 | `if (!res.ok)` 判断，非 2xx 响应抛出明确错误 |
| 新增错误 UI 展示 | 捕获错误后在页面显示红色错误框，列出错误信息和可能原因，引导用户排查 |
| 新增渲染层 try-catch | (第二次修复追加) 区分渲染崩溃和网络错误，显示原始JSON |

## 潜在 CORS 问题说明

即使上述问题全部修复，从浏览器直接调用 DeepSeek API 仍可能遇到 **CORS（跨域资源共享）** 拦截：

- 浏览器出于安全策略，会阻止本地文件向 `https://api.deepseek.com` 发起请求
- 如果 DeepSeek 的 API 服务器没有返回 `Access-Control-Allow-Origin: *` 头，`fetch` 会被浏览器直接拦截
- 此时 `catch` 块会捕获到 `TypeError: Failed to fetch` 或类似的 CORS 错误
- **唯一的正确解法**是搭建后端代理，让服务端转发 API 请求

修复后的代码已包含此场景的错误提示，用户可在错误框中看到具体原因。

## 2026年6月28日：Board 状态切换防抖与跨栏线性动画

### 修改文件

- `app.js`
- `style.css`

### 修改内容

- 新增统一的 Board 稳定渲染流程，在任务状态变化前保存 Done、Today、Upcoming 三个列表各自的滚动位置和卡片坐标。
- 数据更新并重新生成卡片后，在浏览器绘制下一帧前恢复三个列表的 `scrollTop`，避免栏目先跳到顶部再弹回原位置。
- 状态切换期间关闭三栏无关卡片的 `card-enter` 入场动画，防止全量重绘被误看成页面刷新。
- 将原本仅处理删除来源栏的补位逻辑扩展为整个 Board 的 FLIP 布局补间；只有位置实际变化的卡片执行直线平移动画。
- 删除确认完成后改用稳定渲染流程，因此另外两个未参与删除的栏目不再发生滚动或入场动画。
- 重构 Board 上的“完成任务”和 Done 栏“恢复任务”：原卡片先保留不可见占位，再压缩为 3px 黑色直线，平移到目标栏目并展开为目标卡片。
- 完成任务的目标固定为 Done；恢复任务根据 DDL 自动进入 Today 或 Upcoming。
- 线条移动期间不改变三个任务列表的滚动位置；目标卡片暂时不可见但保留布局位置，动画结束后再显示。
- 源栏目后续卡片通过 FLIP 直线向上补位，目标栏目原有卡片按新布局直线移动，不使用旋转、曲线或圆角。
- 对目标卡片位于目标栏可视区域之外的情况增加边界处理：动画在目标栏最近的可视边缘展开，同时不强制滚动目标栏。
- 动画期间锁定重复的完成、恢复和删除操作，并保留系统“减少动态效果”兼容。

### 修改原因

- `renderBoard()` 会重建三个列表的 `innerHTML`，浏览器因此把三个列表的滚动位置重置为顶部，并让全部任务卡片重新播放入场动画。
- 原完成/恢复只给源卡片增加一次水平位移，随后直接全量重绘，缺少卡片移动到目标栏的连续视觉关系。
- 原删除补位只恢复来源列表的滚动位置，没有保护另外两个列表。

### 验证结果

- `node --check app.js` 通过，JavaScript 无语法错误。
- 检查确认删除路径已调用 `renderBoardPreservingLayout()` 和 `animateBoardReflow()`，三栏统一保存和恢复滚动状态。
- 检查完成/恢复路径已统一调用 `startTaskTransfer()`，并根据任务新状态查找实际目标卡片。
- 检查稳定重绘生成的全部 Board 卡片均附加 `skip-card-entry`，状态动画结束后不会再次触发下落入场动画。
- 临时 Edge 自动化实例因本机浏览器调试策略和 GPU 沙箱错误未能启动，相关临时测试目录已清理；本轮未将自动浏览器视觉验收记为通过。
